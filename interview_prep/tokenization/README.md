# Токенизация (Tokenization)

## Ключевые концепты

### Зачем нужна токенизация?
- Нейросети работают с числами, не с текстом
- Word-level словарь слишком большой (сотни тысяч слов + OOV)
- Character-level: маленький словарь, но очень длинные последовательности
- **Subword tokenization** — компромисс: ~30K-130K токенов, редкие слова разбиваются на части

### BPE (Byte Pair Encoding)
**Алгоритм обучения:**
1. Инициализация: все уникальные символы (или байты) как токены
2. Подсчёт частоты всех соседних пар токенов в корпусе
3. Самая частая пара → новый токен, добавляем в словарь
4. Заменяем все вхождения этой пары в корпусе
5. Повторяем до достижения нужного vocab_size

**Byte-level BPE** (GPT-2, GPT-4, LLaMA):
- Начинаем с 256 байт (не символов Unicode)
- Никогда нет OOV — любой текст можно закодировать
- CJK символы = 3 байта → нужно больше merge операций → выше fertility

**Inference (токенизация текста):**
- Greedy longest match: от левого края ищем самый длинный токен из словаря
- Результат зависит от словаря (merge rules), а не только от алгоритма

### WordPiece (BERT, DistilBERT)
- Похож на BPE, но выбирает пары не по частоте, а по **максимизации likelihood** корпуса
- Формула: score = freq(AB) / (freq(A) * freq(B))
- Подслова маркируются `##prefix` (не первое подслово слова)

### Unigram LM / SentencePiece (T5, mT5, ALBERT)
- Начинает с большого словаря, итеративно **удаляет** токены
- Оставляет токены, удаление которых минимально снижает likelihood
- Probabilistic: один текст может иметь несколько токенизаций
- Не требует пре-токенизации по пробелам → лучше для агглютинативных языков

### TikToken (OpenAI)
- Byte-level BPE с регулярным выражением для пре-токенизации
- Пре-токенизация: разбивает текст на "слова" + пунктуацию/пробелы/цифры до BPE
- cl100k_base (GPT-4): 100,277 токенов

## Типичные вопросы на интервью

### Q: Почему две модели дают разное количество токенов на одном тексте?
**A:** Три причины:
1. **Размер словаря**: больший vocab → больше merge операций → более длинные токены → меньше токенов
2. **Обучающий корпус**: словарь обучен на разных распределениях языков → разные merge rules
3. **Алгоритм пре-токенизации**: разные regex паттерны дают разный начальный split

### Q: Как BPE работает с emoji и CJK?
**A:** CJK символы = 3 байта в UTF-8, emoji = 4 байта. При byte-level BPE начинаем с этих байт. Для их слияния нужно больше итераций, чем для ASCII. При фиксированном vocab_size → меньше места на сложные merge → выше fertility для CJK/emoji.

### Q: Что такое fertility токенизатора?
**A:** Fertility = среднее количество токенов на слово для данного языка/текста. Низкая fertility = хорошо (эффективная токенизация). Английский обычно ~1.3-1.5, китайский в модели с английским корпусом может быть 2-4.

### Q: Как решается проблема OOV?
**A:** В byte-level BPE — никак: любой байт есть в начальном словаре. В WordPiece/Unigram — `[UNK]` токен для неизвестных символов (редко в modern tokenizers).

### Q: Почему числа плохо токенизируются?
**A:** BPE обучается на частотности → редкие числа разбиваются побайтно/посимвольно. "1234567" → ["12", "345", "67"] зависит от обучающего корпуса. Это мешает арифметике.

## Частые ошибки на интервью
- ❌ Путать "словарь токенизатора" и "embedding матрицу модели"
- ❌ Считать что BPE всегда начинает с символов (нет — byte-level BPE начинает с байт)
- ❌ Говорить что OOV не бывает в WordPiece (бывает для нестандартных символов)
- ❌ Забывать про пре-токенизацию (split by spaces) перед BPE в TikToken
